# API 形态与 BFF

## 1. REST

**优点：** HTTP 语义清晰；缓存友好（GET）；调试工具普及。  
**痛点：** 多端字段需求差异导致 **过度获取 / 多次往返**。

**实践：** 版本化路径或 Header；错误体结构化；分页一致。

## 2. GraphQL

**优点：** 客户端按需字段；聚合潜力。  
**痛点：** N+1 查询（需 DataLoader）、复杂度攻击（depth/query cost limit）、缓存模型较 REST 复杂。

**适用：** 多端字段差异极大且团队具备治理能力。

## 3. gRPC

**优点：** 二进制高效；强 schema（Protobuf）；适合 **内部服务间** 调用。  
**痛点：** 浏览器原生不支持；需 gRPC-Web + Envoy 等网关。

## 4. BFF（Backend for Frontend）

**动机：** 移动端与 Web 对 **聚合、字段裁剪、协议** 的需求不同；避免单体后端为所有客户端妥协。

| 原则 | 说明 |
|------|------|
| **编排非堆砌业务** | 领域规则仍在核心域服务；BFF 薄聚合 |
| **独立部署** | 可按端发布频率拆分 |
| **鉴权聚合** | 验证令牌、拆分权限声明 |

## 5. 移动端特别考量

- **弱网**：减少往返；合并接口；压缩与增量同步策略。  
- **长版本周期**：字段 **additive** 优先；必填变更走 [兼容策略](../platform-contracts/environments-and-api-compat.md)。

## 6. 契约测试

消费者驱动契约（Pact 等）在 CI 阻止破坏性变更合并。
