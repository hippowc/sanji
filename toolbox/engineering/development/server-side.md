# 服务端

## 1. API 设计要点

- **资源建模**：RESTful 资源名词化；错误模型统一（HTTP 状态码 + 业务错误码）。  
- **分页与筛选**：游标分页优于深度 offset（大数据集）；字段稀疏可用 `fields=`。  
- **版本化**：路径或 Header；与 [API 兼容](../platform-contracts/environments-and-api-compat.md) 联动。

## 2. 幂等与重试

| 场景 | 做法 |
|------|------|
| **写操作重复提交** | `Idempotency-Key` Header + 服务端短期去重表 |
| **消息消费** | 业务幂等键或「恰好生效一次」的状态机 |
| **下游重试** | 指数退避 + jitter；区分可重试错误（5xx、超时）与不可重试（4xx 语义错误） |

避免 **重试风暴**：客户端、网关、服务网格同时对同一故障源重试。

## 3. 超时、熔断与隔离

- **超时链**：客户端 ≤ 网关 ≤ 服务 → 下游；禁止无限等待。  
- **熔断**：连续失败打开断路器；半开探测恢复（Resilience4j、istio 等层次不同）。  
- **舱壁**：线程池/连接池隔离关键依赖，防止拖垮全局。

## 4. 后台任务与队列

- **同步路径**：用户请求应快速返回；重活 **异步化**。  
- **队列**：削峰、解耦、重试；消费端 **幂等**。详见 [消息与异步](../data/messaging.md)。

## 5. 健康检查

| 类型 | 检测内容 |
|------|----------|
| **Liveness** | 进程是否存活；失败则重启 |
| **Readiness** | 依赖（DB、缓存）是否可用；失败则从负载均衡摘除 |

勿将「外部第三方偶尔波动」直接绑定 readiness（可导致不必要的全摘除），可分层降级。

## 6. BFF 与聚合

移动端可能需要 **一次请求聚合多资源**；BFF 承担编排、协议转换、鉴权聚合。避免在 BFF 堆积核心业务规则（防腐败层边界）。
