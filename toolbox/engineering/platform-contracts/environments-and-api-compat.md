# 环境晋升与 API 兼容

## 1. 环境划分建议

| 环境 | 目的 | 与生产的接近度 |
|------|------|----------------|
| **dev** | 个人与日常联调 | 低；可用本地或共享开发集群 |
| **staging** | 上线前集成验证、回归 | **高**：拓扑与权限模型尽量接近 prod；数据 **脱敏** |
| **prod** | 对外服务 | 变更纪律最严 |

**原则：** staging 存在的意义是 **发现问题**；若与 prod 差异过大（如无 TLS、无限制数据库权限），则集成测试可信度下降。

## 2. 晋升流程（概念）

典型晋升链：**dev → staging → prod**。每一步应满足：

- 流水线门禁通过（测试、扫描、制品晋级）。  
- 数据库迁移策略已评审（扩展/收缩兼容）。  
- 配置差异已文档化（开关默认值、限流阈值）。  
- 回滚路径明确（应用版本；迁移是否可逆单独说明）。

详见 [数据库迁移与发布](../cicd/db-migrations-and-release.md)。

## 3. API 向后兼容（尤其移动端）

移动端 App **版本碎片化严重**：用户可能长期停留在旧版。后端变更需默认假设 **多版本客户端并存**。

### 3.1 兼容策略

| 策略 | 适用 |
|------|------|
| **扩展优先** | 新增可选字段、新接口路径；旧客户端忽略未知字段 |
| **废弃窗口** | 标记废弃 → 文档公告 → 至少一个 **主版本周期** 后再移除 |
| **强制升级** | 仅用于安全漏洞或协议不可兼容；需应用内提示与应用商店策略配合 |

### 3.2 版本化方式（择一或组合）

- **路径版本**：`/v1/users`、`/v2/users`，清晰但路由膨胀。  
- **Header 协商**：`Accept: application/vnd.xxx.v2+json`，URL 干净但对网关与文档要求高。  
- **契约测试**：消费者驱动的契约（如 Pact）在 CI 中验证 **破坏性变更不可合并**。

### 3.3 BFF 的角色

对移动端暴露 **稳定的聚合接口**，内部可拆解为多个微服务调用；减少 App 直接感知后端拆分节奏。详见 [API 形态与 BFF](../data/api-and-bff.md)。

## 4. 配置与特性开关

- **远程开关**：紧急关闭某功能而不发版（Kill Switch）；须有权限与审计。  
- **默认安全**：新功能默认关闭；staging 先行开启验证。
