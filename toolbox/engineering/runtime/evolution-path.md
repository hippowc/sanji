# 从小到大演进路径

下列阶梯 **不是唯一顺序**，用于对照：**当下处在哪一级、下一级的触发条件是什么**。

## 阶梯 0：概念验证（可不计入生产）

单实例 VM 或本地 Docker；无 SLAs；**禁止承接付费用户流量**。

---

## 阶梯 1：托管「单体式」计算 + 托管入口

**形态：** 单个托管 Web 应用（各云 App Engine、Elastic Beanstalk、Container Instances 简化模式等）或 **单副本托管容器** + **托管负载均衡（HTTPS）**。

**具备：** 健康检查、自动重启、托管 TLS、日志采集钩子。

**适用：** 极低流量、初期产品验证；团队几乎无运维人力。

**短板：** 垂直扩容有限；发布粒度粗；多服务协作不便。

**升级触发：** 需要 **多实例水平扩展**、独立扩缩后台 worker、或 **蓝绿/金丝雀**。

---

## 阶梯 2：水平扩展的无状态服务层

**形态：** 自动伸缩组 / 托管容器服务的 **多副本**；会话外置（JWT 或 Redis）；静态资源走对象存储 + CDN。

**具备：** 滚动升级；就绪探针；最低限度自动伸缩策略。

**适用：** 典型 Web/API 业务主流阶段。

**升级触发：** 服务数量增多；配置与密钥爆炸；环境一致性难以 shell script 维护。

---

## 阶梯 3：托管 Kubernetes（ACK / EKS / GKE / AKS 等）

**形态：** 节点池 + Deployment/StatefulSet + Ingress/网关 + HPA。

**收益：** 统一编排多服务；声明式配置；生态丰富（网关、Mesh、Operator）。

**成本：** 控制平面与学习曲线；节点安全补丁；集群升级节奏。

**触发信号（满足多条再认真评估）：**

- ≥3 个长期生命周期后端服务需要协同发布与观测。  
- 需要 **独立扩缩** worker、cron、有状态组件 Sidecar 模式等。  
- 团队具备 **平台角色**（哪怕 0.5 FTE）维护集群基线。

详见 [托管 Kubernetes 何时值得](./kubernetes.md)。

---

## 阶梯 4：多集群 / 多区域（进阶）

跨可用区已是默认；此处指 **主动多活 / 单元化架构**。超出默认最小生产范畴，需要专门架构项目。

---

## 与数据的衔接

无论哪一阶梯：**OLTP 与缓存尽量托管**；计算层无状态；存储层承担持久化与高可用。
